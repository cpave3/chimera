## ADDED Requirements

### Requirement: Server tracks the active pending question per session

When the agent calls `ask_question`, the system SHALL create a pending question with a unique request ID generated by `newRequestId()` from `@chimera/core`. The server SHALL track at most one pending question per session.

#### Scenario: Pending question is tracked

- **WHEN** the agent calls `ask_question`
- **THEN** the system SHALL generate a request ID via `newRequestId()`
- **AND** SHALL store the pending question keyed by that ID
- **AND** SHALL emit a `question_request` event carrying the ID and question definitions

### Requirement: Partial form state is client-local

While the user fills in a multi-question form, partial answers SHALL stay on the client. The server SHALL receive only the final submission, cancellation, or timeout. No `question_progress` events SHALL be emitted.

#### Scenario: No progress events during navigation

- **WHEN** the user navigates between questions and changes selections
- **THEN** the server SHALL NOT emit any events for those navigations
- **WHEN** the user finally presses Enter
- **THEN** a single `answerQuestion` request SHALL be sent containing all answers

### Requirement: Answers may be modified before final submission

The user SHALL be able to navigate back to a previously answered question and modify the answer. The submitted response SHALL use the most recent answer for each question.

#### Scenario: Answer modified before submit

- **WHEN** the user answers question 1 with `"A"`, navigates to question 2, then returns and changes question 1 to `"B"`
- **AND** then submits the form
- **THEN** the submitted answer for question 1 SHALL be `["B"]`

### Requirement: User can cancel a pending question

The user SHALL be able to cancel a pending question. The system SHALL emit `question_resolved` with a `cancelled: true` payload, and the agent SHALL receive a tool result indicating cancellation.

#### Scenario: User cancels via TUI

- **WHEN** a question is pending
- **AND** the user presses Escape in the modal
- **THEN** the client SHALL call `cancelQuestion(requestId)`
- **AND** the server SHALL emit `question_resolved` with `cancelled: true`
- **AND** the agent SHALL receive `'{"answers":[],"cancelled":true}'` as the tool result

#### Scenario: SDK consumer cancels programmatically

- **WHEN** a question is pending
- **AND** an SDK consumer calls `client.cancelQuestion(requestId)`
- **THEN** the same cancellation flow SHALL execute

### Requirement: Pending questions time out after a configurable interval

Each question SHALL have a timeout. The default SHALL be 5 minutes; the agent MAY override per call via an optional `timeoutMs` parameter on `ask_question`. On timeout the system SHALL emit `question_resolved` with `timedOut: true`, and the tool SHALL resolve with `'{"answers":[],"timedOut":true}'`.

#### Scenario: Default timeout fires

- **WHEN** a question has been pending for 5 minutes with no answer
- **THEN** the system SHALL automatically resolve it
- **AND** the agent SHALL receive `'{"answers":[],"timedOut":true}'`

#### Scenario: Per-call timeout override

- **WHEN** the agent calls `ask_question` with `timeoutMs: 30000`
- **AND** the question is unanswered after 30 seconds
- **THEN** the question SHALL be resolved with `timedOut: true`

### Requirement: Agent abort clears the pending question

If the agent run is aborted while a question is pending, the pending question SHALL be cleared, no `question_resolved` event SHALL be emitted, and the tool's promise SHALL reject with the abort reason.

#### Scenario: Abort during pending question

- **WHEN** a question is pending
- **AND** the agent's `AbortSignal` fires
- **THEN** the pending question SHALL be removed from session state
- **AND** the tool's promise SHALL reject

### Requirement: Resolution and cancellation reject duplicates

After a question has been resolved, cancelled, timed out, or aborted, any subsequent `answerQuestion` or `cancelQuestion` call for the same request ID SHALL fail with a clear error. Duplicate resolutions SHALL NOT change agent state.

#### Scenario: Duplicate answer rejected

- **WHEN** a question with ID `q-123` has been answered and resolved
- **AND** a client sends another `answerQuestion("q-123", …)` request
- **THEN** the server SHALL respond with an error indicating the question is no longer pending
